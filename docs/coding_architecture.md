# Coding Module Architecture

## Overview

The `coding` module (`src/chemx/core/coding`) provides a structured, workspace‑backed coding agent that turns natural‑language tasks into a sequence of explicit tool actions. It enforces a strict lifecycle (initialize → plan → act ↔ observe → complete/fail) and coordinates multiple components: an agent, a state machine, action representations, a workspace abstraction, prompt profiles, and a router for dispatching interactive prompts.

## High‑level Architecture

```
User Input
   │
   ▼
┌─────────────────┐      ┌──────────────────────────────────────────────┐
│  CodingSession  │─────▶│  WorkflowRouter                              │
│   .run(input)   │      │  .classify(input) → WorkflowKind              │
└─────────────────┘      └──────────────────────────────────────────────┘
           │                              │
           ▼                              ▼
   ┌───────────────┐          ┌───────────────────────────┐
   │  CodingAgent  │          │  CodingAgent (document)   │
   │  (coding WF)  │          │  (DOCUMENT_WORKFLOW)      │
   └───────────────┘          └───────────────────────────┘
           │                              │
           ▼                              ▼
   ┌───────────────┐              shared pattern:
   │  CodingLoop   │           initialize → plan → act ↔ observe
   │  (state m/c)  │              ↓
   └───────────────┘         complete / fail
           │
           ▼
   ┌──────────────────────────────────────────────────────────────────┐
   │  CodingWorkspace (Protocol)                                       │
   │    .inspect(task) → overview                                      │
   │    .execute(action) → ActionResult                                │
   │    .changes() → diff                                              │
   │  LocalWorkspace (implementation)                                  │
   │    aggregates List/Read/Write/Edit/Search/Command/Git tools       │
   └──────────────────────────────────────────────────────────────────┘
```

- **`CodingSession`** – entry point; classifies input via `WorkflowRouter` and delegates to the coding agent or a separate document agent.
- **`CodingAgent`** – extends base `Agent`; holds a model, a `WorkflowProfile`, a `CodingLoop`, and a progress callback. It orchestrates the run lifecycle.
- **`CodingLoop`** – deterministic state machine with phases (`CodingPhase`) and a `CodingRun` dataclass that carries all ephemeral state.
- **`CodingPlan`** – a non‑executable natural‑language plan generated by the model or supplied by the user.
- **`CodingAction`** / `ActionResult` – structured, validated tool actions and their execution outcomes.
- **`CodingWorkspace`** – protocol that decouples action execution from the environment; implemented by `LocalWorkspace` using a suite of tools (`ListTool`, `ReadTool`, `EditTool`, etc.).
- **`WorkflowProfile`** – defines prompt builders and lifecycle callbacks for coding and document workflows.
- **`WorkflowRouter`** – classifies user intent into conversation, coding, or document workflows.

## Key Design Decisions

### `CodingAgent`
- Holds a `model` (LLM), a `system_prompt`, a `WorkflowProfile`, a `CodingLoop`, and an optional `progress_output` callback.
- **Entry points**: `run(user_input)` for conversation; `run_workflow(task, workspace)` for full coding cycles; `run_plan` / `run_actions` for starting from a given plan or explicit actions.
- Internal `_execute_model_loop` runs the main cycle using the state machine, handling planning, action selection, observation compaction, and terminal completion.
- Uses `_select_action` to parse model output into a `CodingAction`; validates the action before execution.
- Compacts observations via `_compact_observations_if_needed` when the context grows large, using deterministic summaries.

### `CodingLoop` & `CodingRun`
- **`CodingRun`** is a frozen dataclass recording `task`, `phase`, `step`, `context`, `plan`, `results` (list of `ActionResult`), `observation_summary`, and terminal info (`response` / `failure_reason`).
- **`CodingLoop`** exposes phase transitions: `start` → `begin_planning` → `set_plan` → `begin_action` / `record_result` → `complete` / `fail`. Each method validates the current phase and raises `ValueError` on invalid transitions.
- Lifecycle is strictly: `INITIALIZE → PLAN → ACT ↔ OBSERVE → COMPLETE | FAILED`. Terminal states cannot be resumed.

### `CodingAction` & `ActionResult`
- **`ActionKind`** enum lists all supported workspace actions: `LIST_FILES`, `READ_FILE`, `SEARCH_TEXT`, `REPLACE_TEXT`, `CREATE_FILE`, `WRITE_FILE`, `RUN_COMMAND`, `BASH`, `GIT_STATUS`, `FINISH`.
- `CodingAction` is a frozen dataclass with optional fields per kind; enforces required fields per kind via `validate()`.
- `parse_action()` extracts a single action from model output, handling markdown fences and fallback JSON extraction.
- `ActionResult` pairs a `CodingAction` with a success flag and the tool output string.

### `CodingPlan`
- Simple dataclass with `text` and `source` (`PlanSource.MODEL` or `USER`). Validated to be non‑empty.
- Designed to be generated once at the start of a workflow and referenced during action selection.

### Workspace (`CodingWorkspace` protocol & `LocalWorkspace`)
- **Protocol** requires `inspect(task)`, `execute(action)`, and `changes()`.
- **`LocalWorkspace`** implements it with a bounded root directory, using `WorkspacePaths` to prevent path traversal.
- Aggregates common tools (`ListTool`, `ReadTool`, `WriteTool`, `EditTool`, `SearchTool`, `CommandTool`, `BashTool`, `GitTool`).
- `execute` selects a handler based on `ActionKind`; catches OSError/RuntimeError/ValueError and returns a failed `ActionResult`.
- `changes()` delivers a Git diff when available; otherwise computes a file‑hash‑based summary of created/modified/deleted files.
- **Safety**: command approval is delegated to a `CommandApproval` callback; Bash is disabled by default and requires explicit opt‑in.

### Workflows & Router
- **`CODING_WORKFLOW`** and **`DOCUMENT_WORKFLOW`** are frozen `WorkflowProfile` instances that specify a system prompt and builders for plan, action, and summary prompts.
- **`DocumentLifecycle`** (via lifecycle factory) enforces the document workflow’s strict ordering: research → draft/review → finish, with no unrelated reads after drafting.
- **`WorkflowRouter`** uses the model to classify user input into `CONVERSATION`, `CODING`, or `DOCUMENT` routes, enabling `CodingSession` to dispatch appropriately.

## The Main Loop

The core execution cycle lives in `CodingAgent._execute_model_loop` and follows these phases, driven by `CodingLoop`:

### 1. Initialisation / Preparation
- `CodingRun` is created via `CodingLoop.start(task)` (phase `INITIALIZE`).
- Workspace is inspected (`workspace.inspect(task)`) to produce an initial context string.
- `begin_planning(run, context)` transitions to `PLAN`.

### 2. Planning
- The model receives the `build_plan_prompt(task, context)` prompt and returns a natural‑language plan.
- The plan text is wrapped in a `CodingPlan` (source `MODEL`).
- `set_plan(run, plan)` moves phase to `ACT`.

### 3. Action Loop (ACT ↔ OBSERVE)
Repeats until termination:
- **Begin action**: `begin_action(run)` increments `step`, stays in `ACT`.
- **Prompt construction**: `build_action_prompt(task, plan, observations, policy, observation_summary)` assembles the current prompt, including compacted context if available.
- **Action selection**: model must output exactly one JSON `CodingAction`; parsed via `parse_action` and validated.
- **Execution**: `workspace.execute(action)` returns an `ActionResult`.
- **Observation recording**: `record_result(run, result)` transitions to `OBSERVE`.
- **Observation compaction**: when the result list exceeds a threshold, older results are compressed into `observation_summary` using `build_observation_compaction_prompt` and the model (generates a factual, action‑oriented summary).
- **Termination check**: if the action kind is `FINISH`, the loop exits (the result output becomes the completion response). If `max_steps` is exceeded, the run fails.

### 4. Completion / Failure
- **Success**: `complete(run, response)` sets phase to `COMPLETE`. A summary prompt (`build_summary_prompt`) asks the model to produce a final summary based on observations and workspace diff. The summary is returned as the agent’s output.
- **Failure**: `fail(run, reason)` sets phase to `FAILED`; an error message is returned.

### Error and Lifecycle Handling
- All phase transitions are guarded: an invalid transition raises `ValueError`.
- `CodingAgent` catches exceptions from the model or workspace and triggers a failure with a descriptive reason.
- Lifecycle callbacks (e.g., `DocumentLifecycle`) can enforce additional constraints by monitoring the `CodingRun` state.

## Data Flow

1. **Input** arrives as a plain string (`user_input`).
2. **Routing**: `WorkflowRouter.classify` returns a `WorkflowRoute` with `objective` and `kind`.
3. **Agent setup**: `CodingSession` selects the appropriate `CodingAgent` (coding or document) and calls its `run_workflow(objective, workspace)`.
4. **Loop state**: `CodingRun` (frozen, replace‑on‑each‑transition) carries all transient state. Each transition returns a new `CodingRun`.
5. **Prompts**: built by `WorkflowProfile` builders; they incorporate task, plan, recent observations (with compaction), and context policy.
6. **Model interaction**: the agent sends a single `Message` per step and parses the reply into a `CodingAction`.
7. **Workspace interaction**: `CodingAction` → `LocalWorkspace.execute` → tool calls; results flow back as `ActionResult`.
8. **Observation compaction**: periodically, the model compresses older `ActionResult` objects into a string stored in `run.observation_summary`, reducing prompt size.
9. **Terminal output**: a final model‑generated summary string (or error) is returned to the user.
